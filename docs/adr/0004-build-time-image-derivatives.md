# Build-Time Generated Image Derivatives

Jukkai generates every web image as a build-time Derivative from a durable Master and serves it as a static asset on Cloudflare Pages. The marketing site does not use a runtime image service (Cloudflare Images, imgproxy, or similar). This applies the same reasoning as ADR-0003 for fonts: the public site behaves like a normal self-hosted static asset consumer, and asset production is a deterministic build step rather than a request-time service.

A runtime image service earns its keep only when the image set is unknown at build time or must appear on the live site within seconds of an edit without a rebuild. Neither is true for Jukkai: the image set is curated and bounded (dozens of project and gallery images, not thousands), and a publish that rebuilds and redeploys in a couple of minutes is acceptable. Building derivatives ahead of time also avoids the request-time transform latency on a cache miss and removes an operational surface Jukkai has already been bitten by. Record the runtime image service as a deferred fallback, reopened only if an instant-publish requirement or an unbounded/unknowable image set appears.

Masters are the archival originals and live in durable storage (Garage S3) outside the site repository, never in git, because git keeps every version of every binary forever. Derivatives are disposable and always reproducible from their Master, so they are safe to regenerate and safe to cache aggressively; content-addressed filenames make stale cache states impossible.

For the launch phase, the tooling is deliberately minimal. Curated web-source exports are committed to the marketing app's assets and `astro:assets` (`<Picture>`) generates the responsive, content-hashed derivative set at build. No bespoke image pipeline is built yet. The existing Python baker remains only for bespoke image synthesis, such as the composited hero field, which is image creation rather than resizing and which `astro:assets` cannot express. A custom derivative pipeline that pulls Masters from Garage S3 and generates web-source with a library like Sharp is introduced only when web-source images can no longer be reasonably committed, or when the source of truth moves out of the repo into the Portal-authored content model.

This decision governs image production only. Delivery of the built site stays as ADR-0001 describes: the marketing app is a static Cloudflare Pages target. The July 2026 reset removed the exploratory publishing-pipeline design. Any future authoring shape must preserve this build-time delivery boundary unless a later ADR explicitly reopens it.
